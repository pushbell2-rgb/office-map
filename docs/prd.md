## 1. Executive Summary

서울숲 사옥 LF층은 **44개의 회의실**이 GC 공용·유비케어·라운지·프로젝트룸·서버실·임원실 등 6가지 타입으로 분산 배치되어 있다. 사무실 이전 후 직원들은 매일 회의 전 5~10분을 회의실 탐색에 낭비하고, 동료 위치 파악을 위해 Slack 메시지를 주고받는다.

본 서비스는 **실제 도면을 3D로 시각화**하고 **Socket.io 기반 실시간 위치 공유**를 결합하여 이 두 문제를 동시에 해결한다. 설치 없이 브라우저에서 바로 사용 가능하며, 단일 서버로 즉시 배포할 수 있다.

## 2. 문제 정의 및 배경

### 2-1. 맥락 및 정량 데이터

서울숲 사옥 LF층 이전 후 직원 설문(n=8, 2026-03-10)과 Slack 로그 분석에서 도출:

| # | 불편 사항 | 측정값 | 연간 비용 환산 |
|---|-----------|--------|-------------|
| 1 | 회의실 위치 탐색 시간 낭비 | 평균 **6.2분/회** (σ=2.1분) | 10,333시간/년 (200명 × 2회/일 × 250일) |
| 2 | Slack 위치 확인 대기 | 평균 **5.3분** 응답 대기 | 13,250시간/년 (200명 × 3회/일 × 250일) |
| 3 | 신규 입사자 공간 적응 | 평균 **1.4주** 소요 | 온보딩 효율 저하 |
| 4 | 회의 지각 경험 | 응답자의 **75%** (6/8명) | 신뢰·협업 문화 저하 |

**총 낭비**: 연 약 **23,583시간** (인당 약 118시간/년)

### 2-2. 문제 구조

```
[신규 사무실 이전]
       │
       ├─ 회의실 번호가 낯설다 (M-1~M-24, P-1~P-4, S-1~S-9 등)
       │        └─ 현장 탐색 필요 → 회의 전 5~10분 낭비
       │
       └─ 동료 위치를 모른다
                └─ Slack 핑 → 응답 대기 → 비효율
```

### 2-3. 핵심 가설

> **"3D 인터랙티브 도면과 실시간 위치 공유를 도입하면, 회의실 탐색 시간을 80% 이상 단축하고 위치 확인을 위한 Slack 메시지를 50% 이상 감소시킬 수 있다."**

## 3. 사용자 분석

### 3-1. 주 사용자

- **대상**: LF층 근무 직원 (GC 소속 + 유비케어 소속)
- **규모**: 약 100~200명 (층 수용 인원 기준)
- **기술 수준**: 일반 직장인, 웹 브라우저 사용에 익숙

### 3-2. 페르소나

| 구분 | 특성 | 핵심 니즈 |
|------|------|---------|
| **신규 입사자** | 사무실 구조 미숙지, 반복 질문 부담 | 스스로 회의실 찾기 |
| **회의 주최자** | 참석자에게 위치 안내 필요 | URL 공유로 위치 설명 대체 |
| **팀장/협업자** | 팀원 현재 위치를 빠르게 파악 | Slack 없이 실시간 위치 확인 |

### 3-3. 사용 시나리오 (Top 3)

1. **회의 5분 전**: 검색창에 "덴버" 입력 → 위치 확인 → 이동
2. **팀원 찾기**: 지도에서 동료 핀 확인 → [→ 이동] 클릭 → fly-to
3. **온보딩**: 신규 입사자가 전체 층 구조를 3D로 파악

## 4. 솔루션 개요

### 4-1. 서비스 개요

실제 도면 이미지를 3D 공간의 바닥 텍스처로 활용하고, 44개 회의실을 타입별 색상의 반투명 3D 박스로 표현한다. 사용자는 이름을 입력해 실시간 세션에 참여하고, 도면 위 원하는 위치를 클릭해 자신의 핀을 설치한다. 다른 사용자의 핀이 실시간으로 동기화되어 3D 공간 위에 표시된다.

### 4-2. 사용자 플로우

```
접속
  └─ 이름 입력 → [입장]
       └─ 3D 지도 화면
            ├─ [검색] 회의실 이름/ID 입력 → 필터링 → 클릭 → fly-to
            ├─ [3D 박스 클릭] → 회의실 정보 패널 → 카메라 이동
            ├─ [내 위치 설정] → 도면 클릭 → 핀 설치
            └─ [동료 패널] → 동료 이름 클릭 → fly-to
```

## 7. 기술 아키텍처 요약

```
[브라우저]
  ├── Three.js (CDN importmap, r161)
  │     ├── OrbitControls    — 마우스 카메라 조작
  │     ├── CSS2DRenderer    — 회의실 이름 라벨
  │     └── Raycaster        — 클릭 인터랙션
  └── Socket.io Client      — 실시간 위치 수신/송신
         │ WebSocket
[Express 서버 (Node.js)]
  ├── 정적 파일 서빙 (frontend/, public/)
  └── Socket.io Server
        └── In-memory Map<socketId, UserState>
```

- **빌드 도구 없음**: importmap으로 CDN 모듈 직접 로드
- **단일 서버**: 배포 시 Railway/Render에 `npm start` 하나로 완료
- **의존성 최소화**: `express`, `socket.io` 2개만 사용

### 8-2. 차별화 포인트 심층 분석

**① 실제 도면 → 3D 즉시 변환 (구현 장벽 최소화)**

기존 사내 지도 서비스는 지도 좌표 등록, API 연동, 별도 서버 구축이 필요하다. 본 서비스는 도면 이미지(JPG) 한 장을 Three.js 바닥 텍스처로 로드하는 방식으로, **신규 건물·층 추가 시 이미지 교체만으로 즉시 대응**이 가능하다.

```
기존 방식: 도면 → CAD 변환 → 좌표 등록 → API 연동 → 배포 (수 주)
본 서비스: 도면 jpg → public/ 교체 → 재배포 (수 분)
```

**② 실시간 위치 공유 + 3D 시각화 동시 제공**

시장에서 실내 위치 공유와 3D 인터랙티브 도면을 동시에 제공하는 무료 도구는 존재하지 않는다. 유료 솔루션(WhereBy, Microsoft Places)은 월 구독료와 IT 도입 절차가 필요한 반면, 본 서비스는 **URL 하나로 즉시 접속·사용 가능**하다.

**③ 의존성 최소화로 높은 이식성**

`express` + `socket.io` 2개 패키지만으로 동작. Docker, Kubernetes 없이 Railway/Render 무료 티어에서 `npm start` 한 줄로 배포된다. 다른 사옥이나 층에 적용 시 추가 인프라 비용 없음.

### 8-3. 시장 규모 (TAM/SAM)

- **TAM**: 국내 사무용 빌딩 인원 약 500만 명. 사무실 이전/확장 후 공간 탐색 비효율이 보편적 문제
- **SAM**: 100명 이상 규모 기업의 사내 도구 수요 (약 8,000개사 추정)
- **현재 타겟**: 단일 사옥 200명 규모 → 향후 다중 층·사옥 확장 시 전사 도입 가능

### 8-4. 한계 및 보완 방향

| 한계 | 현재 대응 | 향후 개선 방향 |
|------|---------|-------------|
| 서버 재시작 시 위치 초기화 | 인메모리 설계 명시 | Redis 기반 영속성 추가 |
| 모바일 터치 미최적화 | 데스크톱 우선 설계 | Touch 이벤트 처리 추가 |
| 회의실 예약 연동 없음 | 위치 파악에 집중 | Google Calendar API 연동 |
| WebGL 미지원 환경 불가 | Chrome/Edge 안내 | 2D 폴백 모드 추가 |

## 11. 범위 외 (Out of Scope)

아래 기능은 MVP 범위에 포함되지 않으며, 향후 로드맵으로 관리:
